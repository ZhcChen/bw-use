# 浏览器列表可选中与批量删除设计

**目标**

在现有浏览器卡片列表中加入可选中能力，支持对当前筛选结果执行批量删除；批量删除时前端最多并发 10 个删除请求。

**现状**

- 前端列表渲染和交互都在 `public/index.html` 的内联脚本中。
- 后端只提供单条删除接口 `DELETE /api/browsers/:id`。
- 数据默认写入仓库根目录下的 `data/`，测试若直接启动服务会污染现有数据库和 profile 数据。

**方案对比**

1. 前端多选，复用现有单条删除接口
   - 优点：改动最小，风险低，直接接入当前后端删除链路。
   - 缺点：批量结果汇总逻辑在前端。

2. 新增后端批量删除接口
   - 优点：接口语义更完整。
   - 缺点：当前项目收益有限，会增加后端改动面和验证面。

**采用方案**

采用方案 1：前端维护选中集合，批量删除时通过受控并发执行器调用现有单条删除接口，并发上限为 10。

## 交互设计

- 工具栏新增 `全选当前结果`、`清空选择`、`批量删除` 按钮，以及 `已选 N 项` 文案。
- 每张浏览器卡片左侧新增复选框，支持单独勾选。
- `全选当前结果` 只影响当前搜索与分组筛选后的可见列表。
- 原有单条 `删除` 按钮保留。
- 删除确认弹窗复用现有弹窗样式，但支持显示单条或批量数量。
- 批量删除后：
  - 成功项从选中集合中移除；
  - 失败项保留，便于用户重试；
  - 前端汇总成功/失败数量并展示结果。

## 实现设计

### 前端

- 在内联脚本中新增：
  - `selectedBrowserIds` 选中集合；
  - `lastRenderedBrowsers` 当前渲染结果；
  - `deleteMode` / `deleteTargetIds` 用于区分单条与批量确认。
- 抽出一个纯函数模块处理批量删除执行：
  - 输入：待删除 id 列表、并发上限、删除执行函数；
  - 输出：成功列表、失败列表。
- 列表渲染时根据选中状态更新复选框和工具栏。

### 后端

- 将数据根目录改为可配置，默认仍然是仓库内 `data/`。
- 测试时通过环境变量切换到临时目录，隔离 SQLite 与 profile 数据。

## 错误处理

- 批量删除过程中单个请求失败时，不中断整批。
- 前端在完成后提示：
  - 全部成功；
  - 部分失败；
  - 全部失败。

## 测试设计

- 为数据目录解析逻辑编写测试，验证默认目录与环境变量覆盖行为。
- 为批量删除执行器编写测试，验证：
  - 并发上限不超过 10；
  - 成功/失败结果能正确汇总。
- 所有自动化测试都使用临时目录，不读取或写入现有 `data/`。
